--------------------------------------------------------------------------------
1. PROJECT GOAL
--------------------------------------------------------------------------------

AeroView is a Next.js web application that analyses a flight route and tells the
user which side of the airplane (left or right window seat) offers the best views
during the flight. "Best views" is defined by a weighted combination of:
  - Sunrise/sunset crossings visible from one side
  - Solar elevation (night vs. twilight vs. full day)
  - Landmarks, mountain ranges, and cities visible from one side
  - User preference to avoid direct sunlight (avoidSun weight)

The recommendation is backed by:
  1. Great-circle geodesy (not straight-line — actual flight path)
  2. SunCalc solar positioning (precise sun azimuth and elevation per waypoint)
  3. A scoring engine that weights events and produces a SeatVerdict
  4. An LLM narrative (Gemini 2.5 Flash, streaming) that explains the verdict
  5. A Mapbox GL JS 3D animated map with a moving plane icon

4. DATA FLOW — END TO END
--------------------------------------------------------------------------------

[Browser — Client Side]

1. User fills InputForm.tsx
     - Searches airports by IATA / city / name (min 3 chars, max 8 results)
     - Sets departure date + time (stored as UTC ISO string)
     - Sets 4 preference weights: sunrise, sunset, landscape, avoidSun (-2 to 3)
     - Toggles isOvercast (reduces all scores by 80%)

2. ClientApp.tsx receives FlightInput and runs the pipeline:

   a. greatCircleWaypoints(origin, destination, departureUTC)
        → WaypointData[] — one point every 50 km along the great circle
        → Each point has: lat, lon, bearingDeg, cumulativeKm, utcTime
        → Solar fields initialised to 0/null/[]

   b. enrichWaypointsWithSolar(rawWaypoints, pois)
        → WaypointData[] — same array, solar fields filled in:
        → solarAzimuthDeg, solarElevDeg, isHorizonEvent, nearbyPOIs

   c. computeVerdict(enrichedWaypoints, preferences)
        → SeatVerdict { winner, confidence, leftScore, rightScore,
                        leftEvents, rightEvents, flightDurationMin }

3. ResultScreen.tsx renders:
     - SeatBadge (winner + confidence)
     - Two WindowMockup (left + right, highlighted per winner)
     - Two EventList (left events / right events)
     - MapView (dynamic import, ssr:false) — Mapbox 3D animated map
     - NarrativeCard — fetches POST /api/narrative (streaming Gemini text)

[Server — API Route]

4. POST /api/narrative receives { verdict: SeatVerdict }
     - Builds a prompt summarising the route verdict
     - Calls Gemini 2.5 Flash with streaming
     - Returns ReadableStream<string> as text/plain
     - NarrativeCard reads the stream chunk-by-chunk and appends to display

7. LAYER IMPLEMENTATIONS IN DETAIL
--------------------------------------------------------------------------------

=== Layer 2A — geodesic.ts ===

haversineKm(lat1, lon1, lat2, lon2) → number
  Standard haversine formula. Returns great-circle distance in km.

bearingDeg(lat1, lon1, lat2, lon2) → number [0, 360)
  Forward azimuth from point 1 to point 2.
  y = sin(Δλ) * cos(φ2)
  x = cos(φ1)*sin(φ2) - sin(φ1)*cos(φ2)*cos(Δλ)
  return (atan2(y, x) * 180/π + 360) % 360

intermediatePoint(lat1, lon1, lat2, lon2, fraction) → {lat, lon}
  SLERP (spherical linear interpolation), not linear interpolation.
  Produces true great-circle intermediate points.

greatCircleWaypoints(origin, destination, departureUTC) → WaypointData[]
  - Samples every WAYPOINT_STEP_KM along the great circle
  - Antimeridian fix: if consecutive lon diff > 180°, adjust by ±360
  - For each waypoint, computes bearing TOWARD next waypoint
  - CRITICAL FIX: nextPt.lon is also antimeridian-corrected relative to
    current lon before calling bearingDeg (prevents wrong bearings on
    Pacific/antimeridian routes like LAX→SYD)
  - Solar fields initialised to 0/null/[] — enriched in Phase 2

=== Layer 2B — solar.ts ===

toNorthClockwiseDeg(rawAzimuthRad) → number
  SunCalc returns azimuth in radians from SOUTH, clockwise.
  Convert: ((rawAzimuthRad + π) * 180/π) % 360
  This is applied to EVERY SunCalc azimuth read — no exceptions.

getSolarPosition(lat, lon, date) → SolarPosition
  Wraps SunCalc.getPosition(). Returns { solarAzimuthDeg, solarElevDeg }.

detectHorizonEvent(prevElevDeg, currElevDeg) → 'sunrise' | 'sunset' | null
  prevElevDeg < 0 and currElevDeg >= 0 → 'sunrise'
  prevElevDeg >= 0 and currElevDeg < 0 → 'sunset'

elevationMultiplier(elevDeg) → number
  Quality multiplier based on sun position:
  < TWILIGHT_ELEV_DEG → 0 (too dark, not scored)
  < 0                 → proportional 0→1 mapping
  >= 0                → 1.0

scanNearbyPOIs(lat, lon, bearing, pois) → POIResult[]
  Scans pois.json for entries within POI_RADIUS_KM.
  Sorts ascending by distanceKm.
  Caps at MAX_POIS_PER_WAYPOINT.
  Returns array of { poi, distanceKm, bearingFromPlane }.

enrichWaypointsWithSolar(rawWaypoints, pois) → WaypointData[]
  IMPORTANT: Uses a for-loop (NOT .map()) so each iteration reads
  prevElevDeg from the ALREADY-ENRICHED result array. Using .map()
  would read the original solarElevDeg=0 for all prev entries, causing
  false sunrise/sunset detections (especially in polar night tests).

=== Layer 3 — recommend.ts ===

sideOfPlane(planeBearing, objectAzimuth) → 'left' | 'right'
  relativeBearing = (objectAzimuth - planeBearing + 360) % 360
  [0, 180)  → 'right'  (object is to the right of the plane)
  [180, 360) → 'left'  (object is to the left of the plane)

scoreSolarEvent(type, elevDeg, preferences) → number
  score = weights[type] * elevationMultiplier(elevDeg) * weatherFactor

scoreLandmark(distanceKm, preferences) → number
  Decreasing score as distance increases.
  Beyond POI_RADIUS_KM → 0.

computeVerdict(enrichedWaypoints, preferences) → SeatVerdict
  Iterates all waypoints:
    - Horizon events → scoreSolarEvent, sideOfPlane, push to leftEvents/rightEvents
    - avoidSun → penalises the side the sun is on (not bonus to other side)
    - POIs → scoreLandmark, sideOfPlane, push to leftEvents/rightEvents
  POI DEDUP FIX: seenPoiIds = new Set<string>() — skips any POI that already
    appeared in events. Each POI contributes exactly ONCE regardless of how
    many consecutive waypoints it's within POI_RADIUS_KM of. This prevents
    landmarks like Manhattan or Aurora zones from creating 5-8 duplicate
    markers on the map and over-inflating the score.
  winner='either' when |leftScore - rightScore| / total < LOW_CONFIDENCE_THRESHOLD

=== Layer 4A — MapView.tsx ===

Must be dynamically imported in ResultScreen:
  const MapView = dynamic(() => import('./MapView'), { ssr: false })
  Mapbox GL JS requires browser DOM — will crash on SSR.

Map config:
  style: 'mapbox://styles/mapbox/dark-v11'
  pitch: 50, bearing: -10, antialias: true
  3D terrain: source 'mapbox-dem' + setTerrain({ exaggeration: 1.5 })

Layers (in order, bottom to top):
  1. route-colored  — all segments coloured by solarElevDeg:
                      night(<-6°)='#1e3a5f', twilight='#f97316', day='#facc15'
                      line-width: 3, line-opacity: 0.75
  2. route-dashed   — subtle dashed overlay, line-opacity: 0.45
  3. flown-glow     — wide blurred blue glow for traveled path
  4. flown-core     — crisp light-blue line for traveled path

Markers:
  - Origin: blue Mapbox default marker + popup
  - Destination: indigo Mapbox default marker + popup
  - Verdict events: emoji icons (🌅🌇🏙️📍) positioned at the waypoint closest
    to each event's timeMinFromDeparture. Popup shows name + left/right side.
  - Plane: SVG top-down airplane (nose at top = north), two-layer structure

Plane marker TWO-LAYER structure (critical):
  planeOuter (div) → passed to mapboxgl.Marker({ anchor: 'center' })
    Mapbox writes positioning transform here: translate(Xpx, Ypx)
    NEVER touch planeOuter.style.transform directly.
  planeInner (div, child of planeOuter) → contains the SVG
    We write rotation ONLY here: firstElementChild.style.transform = rotate(θ)

Bearing correction formula:
  θ = wp.bearingDeg - map.getBearing()
  Reason: CSS rotate(θ) is from screen-up, not from north. map.getBearing()
  is the compass direction currently pointing up. Subtracting it keeps the
  plane aligned with the route regardless of map rotation by the user.
  map.on('rotate', ...) listener re-applies this correction whenever the
  user rotates the map. Uses currentIndexRef (not state) to avoid stale closures.

Time-of-day fog (fogForElev):
  solarElev < -12°: near-black sky, stars 95%
  -12° to -6°:      dark blue, stars 65%
  -6° to 0°:        warm orange twilight, stars 20%
  0° to 15°:        golden morning/evening sky
  > 15°:            bright blue day sky, no stars
  Called on every currentIndex tick via map.setFog().

Animation:
  setInterval every 80ms → setCurrentIndex(prev + 1)
  On each currentIndex change:
    planeRef.setLngLat([wp.lon, wp.lat])  — moves plane
    planeInner.style.transform = rotate(bearingDeg - mapBearing)  — rotates plane
    flownSource.setData(waypoints.slice(0, currentIndex+1))  — grows flown path
    map.setFog(fogForElev(wp.solarElevDeg))  — updates sky

=== Layer 4B — /api/narrative/route.ts ===

POST /api/narrative
  Receives: { verdict: SeatVerdict }
  Validates with isValidRequest() before calling Gemini
  Model: gemini-2.5-flash
  Returns: new Response(ReadableStream, { 'Content-Type': 'text/plain' })
  Error: 400 for bad request, 500 for Gemini failure

NarrativeCard.tsx:
  fetch('/api/narrative', { method: 'POST', body: JSON.stringify({ verdict }) })
  Uses response.body.getReader() to read stream chunks
  Uses cancelled flag to prevent setState after unmount

=== Layer 5 — ResultScreen.tsx ===

Assembles: SeatBadge + WindowMockup (×2) + EventList (×2) + MapView + NarrativeCard
topEvent() helper picks highest-scoring event per side for WindowMockup
MapView has no wrapping height div — it manages its own height (450px map + slider)

=== TimeSlider.tsx ===

FIXED BUG: The visual bar div and the invisible <input type="range"> must share
a common `position: relative` parent. Without it, the `absolute`-positioned input
floats away from the track, making click-to-seek not work.

Correct structure:
  <div className="relative h-4 flex items-center">
    <div className="w-full h-2 ... pointer-events-none">   {/* visual bar */}
      <div style={{ width: `${pct}%` }} />                 {/* fill */}
    </div>
    <input type="range" className="absolute inset-0 ..." />  {/* click target */}
  </div>

pointer-events-none on the visual bar ensures clicks fall through to the input.

=== EventSidePanel.tsx ===

Shows verdict.leftEvents + verdict.rightEvents, sorted by timeMinFromDeparture.
For each event, computes diff = event.timeMinFromDeparture - currentMin:
  diff < -5          → past (grayed, ✓ icon)
  |diff| <= 5        → active ("NOW — look left/right!", highlighted)
  5 < diff <= 30     → upcoming (shows countdown)
  diff > 30          → future (dimmed)

B5 — Gemini model 404 (Phase 6)

B7 — MapView.tsx: plane transform overwrites Mapbox position (Mapbox migration)
  Setting planeEl.style.transform = 'rotate(...)' overwrote Mapbox's

  to have no effect.
  Fix: two-layer element structure — rotate only planeInner (child), never planeOuter.

B8 — MapView.tsx: SVG arrow shape not a plane (Mapbox migration)
  Original path M12,2 L7,15 L10,13 ... created an arrowhead, not an airplane.
  Fix: rewrote SVG path to proper top-down airplane silhouette:
  M12,1 L13,9 L22,13 L13,15 L13.5,21 L12,20 L10.5,21 L11,15 L2,13 L11,9 Z

B9 — recommend.ts: duplicate POI events (multi-waypoint overlap)
  POI_RADIUS_KM=250, WAYPOINT_STEP_KM=50 → same POI in 5+ consecutive
  waypoints → 5+ duplicate ScoredEvents + 5+ map markers + inflated score.
  Fix: seenPoiIds = new Set<string>() — skip if poi.id already seen.

B10 — TimeSlider.tsx: click-to-seek not working
  <input type="range"> had position:absolute but parent had no position:relative.
  Input floated relative to distant ancestor, misaligned from visual track.
  Fix: wrap track + input in a shared position:relative container.
  pointer-events-none on visual bar ensures clicks reach the input.

B11 — MapView.tsx: plane bearing wrong on rotated map
  rotate(bearingDeg) rotates from screen-up, not from north.
  With map bearing=-10°, plane appears 10° off the route.
  Fix: rotate(bearingDeg - map.getBearing())
  Also added map.on('rotate',...) listener using currentIndexRef to update
  rotation whenever user rotates the map (avoids stale closure issue).

B12 — geodesic.ts: antimeridian bearing wrong for Pacific routes
  waypoint lon is antimeridian-adjusted (e.g. -200) but nextPt.lon is raw
  SLERP value (e.g. 170). Δλ = 370° → completely wrong bearing.
  Fix: apply same ±360 antimeridian correction to nextLon before bearingDeg().

12. KEY ARCHITECTURE DECISIONS
--------------------------------------------------------------------------------

1. Server Component for data loading
   page.tsx is a Server Component that imports airports.json and pois.json.
   These stay out of the client JS bundle. Passed as props to ClientApp.

2. ClientApp orchestrates the pipeline
   ClientApp.tsx ('use client') runs all three lib functions in sequence on
   the client. No API routes for geodesic/solar/recommend — pure client-side.
   Only the LLM narrative uses a server API route.

3. Mapbox GL JS instead of Leaflet
   Replaced in later development for 3D terrain, fog/atmosphere, and WebGL
   rendering quality. react-leaflet and leaflet packages still installed but unused.

4. Dynamic import for MapView
   next/dynamic(() => import('./MapView'), { ssr: false }) — Mapbox GL JS
   accesses window/document on import. SSR would crash Next.js build.

5. Streaming LLM narrative
   ReadableStream from the API route allows text to appear word-by-word.
   NarrativeCard reads chunks via response.body.getReader().

6. SunCalc azimuth conversion
   SunCalc returns azimuth from SOUTH, clockwise (radians).
   toNorthClockwiseDeg = ((rawRad + π) * 180/π) % 360
   Applied everywhere without exception.

7. For-loop not map() in enrichWaypointsWithSolar
   Each waypoint's horizon event detection needs the PREVIOUS waypoint's
   already-enriched solarElevDeg. .map() gives original (all 0s). for-loop
   reads from the already-built result array.

8. Two-layer plane marker
   Mapbox GL JS owns the transform of its custom element's root div.
   We must only touch child element transforms for rotation.

9. seenPoiIds dedup in computeVerdict
   Without this, a POI near the route gets 1 ScoredEvent per waypoint (5+
   duplicates for WAYPOINT_STEP=50 and POI_RADIUS=250). Dedup by poi.id
   keeps scoring and map markers clean.

10. Map bearing correction for plane rotation
    CSS rotate() is relative to screen-up. Map bearing offset must be
    subtracted. Also listen to map rotate events for real-time correction.
